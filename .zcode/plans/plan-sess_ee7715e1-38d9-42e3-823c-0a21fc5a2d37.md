# MAL Anime Score Predictor ‚Äî Diagnosis & Implementation Plan

## Diagnosis (root causes of the broken Vercel site)

I've inspected the entire repo and confirmed the failures empirically with live API calls.

**1. Frontend depends on a backend that isn't deployed.** `AnimePredictionsApp.jsx` does `fetch(\`${VITE_API_BASE_URL}/season/.../predictions\`)`. There is no `VITE_API_BASE_URL` set on Vercel, and the FastAPI app (`src/serving/app.py`) is never deployed there. Result: every load throws ‚Üí no scores. The root `vercel.json` builds only the Vite frontend.

**2. Image URLs are never captured.** The local normalized dataset (`data/normalized/anime.parquet`, 3819 rows) is **entirely from AniList** (`source_api == anilist` for all rows), and the AniList‚ÜíJikan item mapping in `client.py` deliberately drops `coverImage`. So neither the parquet nor the predictions contain any image column. The committed `predictions_2025_fall.parquet` has only `[mal_id, title, year, season, pred_score]`. The frontend's per-card Jikan cover fallback (`fetchCoverFor`) hammers `api.jikan.moe` per MAL ID and usually hits 429/rate-limits ‚Üí blank covers.

**3. Jikan season endpoints are currently unstable.** Live testing: `seasons/2026/summer`, `seasons/2024/winter`, and `seasons/upcoming` all return HTTP 504 ("Jikan failed to connect to MyAnimeList"), while single-anime (`anime/1`) works. AniList season queries work and return cover images on a public CDN. The pipeline must therefore robustly prefer Jikan but fall back to AniList, and the existing AniList path must capture images.

**4. Data/model artifacts are gitignored and not reproducible.** `.gitignore` ignores all of `data/`, `*.parquet`, `*.joblib`. Only `data/predictions/predictions_2025_fall.parquet` was force-tracked. `.vercelignore` then excludes `/data/` and `*.parquet` from the deploy anyway, so even the tracked parquet never reaches Vercel. A fresh clone has none of the heavy data ‚Äî by design ‚Äî but has **no way to get the lightweight frontend-ready predictions** without running the whole ML pipeline.

**5. Data leakage risk in current feature set.** `build_features.py` uses `members` and `favorites` as features. These come from the same snapshot as the label `score` (AniList current values), so for finished shows they reflect post-airing popularity ‚Äî a leakage concern the task explicitly flags. The split in `train.py` uses `GroupShuffleSplit` (random) rather than a strict time-ordered holdout, which also leaks future information into validation. (Note: `ingest_details.py`/`labels.parquet` exist but are unused by the training path ‚Äî `label_score` comes from the `score` column in `anime.parquet`.)

## Confirmed decisions (from user)
- **Frontend data source:** static committed JSON only (no backend in production). FastAPI kept for local dev.
- **Seasons shown:** Summer 2026 (new) + 2025 fall (existing).

## Implementation Plan

### Part A ‚Äî Data pipeline (`src/mal/client.py`, `src/ingest.py`)

**A1. Capture image URLs.**
- In `_anilist_to_jikan_item`, add `coverImage.large`/`extraLarge` to a new `images` field shaped like Jikan's (`{"jpg":{"large_image_url":...}, "webp":{"large_image_url":...}}`).
- In `normalize_season_payload`, capture `images` (Jikan already provides it) and `themes`, `producers`, `licensors`, `aired`/`season`/`year` (Jikan returns these; currently dropped).
- Add an `image_url` (string, large webp preferred) column to the normalized store.

**A2. Robust Jikan-with-AniList-fallback.** Keep current behavior but:
- Increase retry patience for the season endpoint (already retries 504/429; add a longer base wait specifically for 504, since MAL upstream 504s are common).
- When Jikan season fails entirely, fall back to AniList **with images** (current code does this but loses images ‚Äî fixed by A1).

**A3. Caching.** Raw payloads are already cached per season in `data/raw/`. Add an `--use-cache` flag to `ingest` so re-runs skip API calls when the raw JSON already exists (speeds up reproducibility; still respects rate limits on fresh fetches).

**A4. A new lightweight command** `src/ingest_season.py` (or extend `predict`/a `generate_predictions` module) that fetches **one** target season (e.g. Summer 2026) via Jikan‚ÜíAniList fallback, normalizes it with images, and returns a clean DataFrame ‚Äî used by the prediction step so we don't depend on a separate full ingest.

### Part B ‚Äî Model (`src/features/build_features.py`, `src/models/train.py`, `src/models/predict.py`)

**B1. Leakage-safe features.** Drop `members`/`favorites` as raw features, OR replace with pre-release-safe proxies. Practically, since our snapshot is "current" AniList data, the only leak-safe features available for an *upcoming* season are: studio, genres, themes, demographics, source, type, rating, episode count, season, year, and title/synopsis length. I'll build features from those only, and document the assumption. (Members/favorites at prediction time would be ~0 for unreleased shows anyway, so they add no signal for the actual target.)

**B2. Richer feature engineering.**
- Studio quality encoding: target-encode top studios by their historical mean score (computed only from train folds to avoid leakage), unknown studios ‚Üí global mean.
- Genre/theme/demographic multi-hot (top-N by frequency).
- Source, type, rating one-hot (existing).
- `episodes` (log1p, missing‚Üímedian), `year`, `season` cyclic or one-hot (existing).
- Title/synopsis length (existing) + simple synopsis sentiment/keyword counts optional.
- A `is_sequel` heuristic if relations contain "Prequel" (leakage-safe ‚Äî known pre-release).

**B3. Proper time-based split.** Replace `GroupShuffleSplit` with a chronological holdout: train ‚â§ 2023, validate = 2024, test = most recent completed season. This matches the real prediction task (predict future seasons from past).

**B4. Multi-model comparison + tuning.** Compare RandomForest (baseline), GradientBoosting/ HistGradientBoosting, and (if installable) LightGBM. Use a small `GridSearchCV`/`RandomizedSearchCV` or manual time-fold CV. Pick best by validation MAE. Save as `model.joblib` with a fitted `ColumnTransformer`/pipeline (so inference applies identical preprocessing). Print MAE/RMSE/R¬≤ for train/val/test before & after, and persist a `metrics.json`.

**B5. Prediction output enrichment.** `predict.py` writes a parquet with the full metadata the frontend needs: `mal_id, title, year, season, pred_score, pred_low, pred_high (uncertainty via quantile or ¬±std across models), image_url, genres, themes, studio, source, synopsis, mal_url, type, episodes, rating`.

### Part C ‚Äî Frontend artifact + website (`src/models/predict.py` or new `src/export_predictions.py`, `anime-frontend/`)

**C1. Generate committed static JSON.** A new script `src/export_predictions.py` reads the prediction parquet(s) and writes JSON to `anime-frontend/public/predictions/summer-2026.json` and `.../fall-2025.json`. Each is a JSON array of objects with all fields the cards need. This is the **only** artifact the frontend reads in production.

**C2. Rewrite `AnimePredictionsApp.jsx`:**
- Remove the API-base-URL input and `fetchPredictions` against a backend.
- Add a season selector populated from a committed `anime-frontend/public/predictions/index.json` (list of available seasons + labels).
- `fetch(\`/predictions/${year}-${season}.json\`)` ‚Äî same-origin, works on Vercel static hosting.
- Remove the per-card Jikan cover fallback (images now come from the JSON). Keep a graceful placeholder if `image_url` is null.
- Show `pred_score`, uncertainty range, genres, studio, source, type, episodes, synopsis (expandable), MAL link.
- Keep search/sort/CSV export.

**C3. `AnimeCard.jsx`:** read `image_url` as a string directly (the new JSON stores a string, not the nested `images` object). Keep the badge logic. Add studio/genres/source line + uncertainty chip.

**C4. `index.html`:** update `<title>` to "MAL Anime Score Predictions".

### Part D ‚Äî Vercel / deployment (`vercel.json`, `.vercelignore`, `anime-frontend/`)

**D1. Root `vercel.json`:** keep framework=vite, build from `anime-frontend/`. Files in `anime-frontend/public/` are copied into the build output by Vite automatically, so the JSON is served at `/predictions/...`. Keep the SPA rewrite. No backend route needed.

**D2. `.vercelignore`:** remove the blanket `*.parquet`/`/data/` exclusions only if needed ‚Äî actually they're fine, because the frontend JSON lives under `anime-frontend/public/` (not `data/`), so it is **not** ignored. Verify `anime-frontend/public/predictions/*.json` is tracked by git (adjust `.gitignore`: the root `.gitignore` ignores `data/` and `*.parquet` but **not** `*.json`, so JSON in the frontend is safe). I'll add an explicit `!anime-frontend/public/predictions/*.json` safety net.

**D3. Delete `anime-frontend/vercel.json`** (duplicate/confusing; root one governs). Or keep it but ensure consistency. I'll keep the root one as the source of truth and remove the frontend-local one to avoid ambiguity.

### Part E ‚Äî Reproducibility & docs (`README.md`, `requirements.txt`, scripts)

**E1. `requirements.txt`:** add `xgboost` or `lightgbm` (I'll try LightGBM; fall back to sklearn `HistGradientBoostingRegressor` if install fails on this env). Pin nothing aggressively.

**E2. README rewrite** with: fresh-clone setup (venv, pip, npm install), the full pipeline commands (ingest ‚Üí build_features ‚Üí train ‚Üí predict ‚Üí export_predictions), how to run the frontend locally (now just `npm run dev`, no `.env.local` needed since data is static), how to deploy to Vercel (just `vercel` ‚Äî no env vars), and a clear statement of what's committed vs. generated.

**E3. Keep FastAPI** for local dev / optional use, but document that production uses static JSON.

**E4. A one-shot repro script** `scripts/reproduce.sh` (and `.ps1`) chaining: ingest 2018‚Äì2025 ‚Üí build_features ‚Üí train ‚Üí predict 2025:fall & 2026:summer ‚Üí export_predictions. With caching, re-runs are fast.

## Limitations / assumptions
- Jikan season endpoints are intermittently 504 due to upstream MAL issues; the pipeline falls back to AniList (which provides images + the same metadata). I'll prefer Jikan where it works.
- Members/favorites are treated as leak-prone and excluded from features for upcoming-season prediction (justified in code comments).
- Model is a practical tabular regressor on metadata; it won't beat MAL's wisdom-of-crowds but should land in a reasonable MAE range (~0.5‚Äì0.7). I'll report actual numbers.
- Heavy training data stays gitignored/local; only the lightweight frontend JSON is committed.
